Always use virtual environment in current project(venv)

## 1. Project Overview
You are tasked with building a supervised machine learning model to specifically detect Ransomware activity. The project will leverage a labeled dataset generated by the Picus Breach and Attack Simulation (BAS) platform. The model will be trained to distinguish between normal operational behavior and the distinct patterns of Ransomware attacks, using QRadar Rule trigger frequencies as its primary features.
The core objective is to create a high-efficacy detector for one of the most critical threats to our organization.

## 2. Key Technical Specifications
Primary ML Algorithm: Random Forest Classifier (sklearn.ensemble.RandomForestClassifier). We are choosing this over MLP for its strong performance on tabular data, interpretability (feature importance), and faster training cycles.
- Python Version: Ensure all code is compatible with Python 3.6.8.
Input Data Schema:
Each row represents a 30-minute aggregated time window.
Features: 2898 columns, each corresponding to a unique QRadar Rule ID. Cell values are the Rule's trigger count. The exact number of features may be refined.
Labels: A binary target column (is_attack), where 1 indicates a window containing a Picus-simulated Ransomware attack, and 0 indicates a normal window.
Core Libraries:
pandas & numpy for data manipulation.
scikit-learn for the Random Forest model, preprocessing, and metrics.
joblib for model persistence.
pymongo for data extraction from MongoDB.
matplotlib for plotting results.

## 3. Required Implementation Modules

The project will be structured into modules for data processing, training, and evaluation.

# Module 1: data_processing.py
This module will contain functions for fetching data and creating the final feature set.
Function: create_feature_vectors(mongo_collection, rule_list)
Purpose: To query labeled event data from MongoDB and aggregate it into feature vectors.

# Parameters:
mongo_collection: A pymongo collection object.
rule_list (list): A list of the 2898 Rule IDs that will form the columns of the DataFrame.
Since data and training are separate, there will be a folder Qradar_rule to store all the rule files Training_data folder to store all the training data.Qradar rule and training data are manually fetched and stored in this folder.

Implementation Steps:
Query MongoDB for event data within defined time windows.
For each 30-minute window, aggregate the trigger counts for every Rule in the rule_list.
Create a Pandas DataFrame where each row is a 2898-dimension feature vector.
Add a label column based on whether the time window corresponds to a Picus Ransomware simulation (1) or normal activity (0).
Return: A labeled Pandas DataFrame ready for training.

# Module 2: model_training.py
This module will handle the training and tuning of the Random Forest model.
Function: train_ransomware_detector(feature_data_path, model_save_path)
Purpose: To train, tune, and save the Random Forest classifier.
Parameters:
feature_data_path (str): Path to the final labeled dataset CSV.
model_save_path (str): Path to save the trained model (e.g., './model/ransomware_detector.joblib').
Implementation Steps:
Load the labeled DataFrame from feature_data_path.
Separate features (X) and labels (y).
Split the data into training and testing sets using train_test_split. Important: Use stratify=y to ensure the proportion of attack samples is the same in both sets due to class imbalance.
Instantiate RandomForestClassifier with these key hyperparameters:
n_estimators=200 (A good starting point)
class_weight='balanced_subsample' (Crucial for handling the highly imbalanced dataset where attack samples are rare)
max_features='sqrt' (Default is often good, but can be tuned)
random_state=42
n_jobs=-1
Fit the model on the training data (X_train, y_train).
Save the trained model to model_save_path using joblib.dump().

# Module 3: model_evaluation.py
This module is for assessing the model's performance and extracting insights.
Function: evaluate_model(model_path, test_data_path, rule_list)
Purpose: To evaluate the trained model and identify the most important features for detecting ransomware.

# Parameters:
model_path (str): Path to the trained .joblib model file.
test_data_path (str): Path to the testing dataset CSV.
rule_list (list): The list of the 2898 Rule IDs.
Implementation Steps:
Load the model and the test data.
Generate predictions on the test set (X_test).
Calculate and print key performance metrics, focusing on:
Confusion Matrix: To see True Positives, False Positives, etc.
Classification Report: Includes Precision, Recall, and F1-Score. Pay close attention to the metrics for the "attack" (label 1) class. Recall is particularly important to minimize missed detections.
ROC AUC Score.
Feature Importance Analysis:
Extract model.feature_importances_.
Create a Pandas DataFrame mapping these importances to the rule_list.
Sort the DataFrame to find the Top 20 most important Rules for detecting ransomware.
Print the Top 20 Rules and their importance scores. This is a key deliverable for the security team.

##  Project Architecture: Unified Data Processing Pipeline

1. Core Design Principles
	•	Maximize Code Reuse:
The training and detection workflows share exactly the same core data processing and feature engineering code paths.
	•	Early Abstraction:
Differences in data source formats (e.g., CSV vs. JSON) are normalized at the very beginning of the data processing stage, ensuring that all subsequent modules operate on a standardized data structure.
	•	Single Responsibility:
Each .py module has a clear, independent function, making it easy to test and maintain.
	•	Eliminate Training–Serving Skew:
Prevent any risk of model performance degradation caused by inconsistencies between training and detection logic.

1. Explanation for Core Pipeline Modules
A. Core Pipeline Modules
	1.	data_loader.py (enhanced responsibilities)

	•	Purpose: Handle all data input and “load-time” preprocessing; unify heterogeneous sources into a standard Pandas DataFrame.
	•	Main function/class: load_data(mode, config)
	•	Used by: Invoked first by pipeline_controller.py.
	•	Details:
	•	mode='train': read CSVs from config['training_data_path'] and config['attack_data_path'].
	•	mode='detect': connect to MongoDB via mongodb_connection.py and query the last 30 minutes.
	•	Timestamp parsing: call time_utils.parse_qradar_timestamp() to convert QRadar strings (e.g., “Jul 29, 2025, 9:50:55 AM”) to datetime.
	•	Type coercion: enforce rule_id and count as int, hostname as str.
	•	Basic missing values: define policy at load stage (e.g., drop rows missing hostname/rule_id or fill defaults).
	•	Column standardization: ensure final schema ['hostname','rule_id','timestamp','count','source_label'] (the last is optional).
	•	Output: Clean, typed, standardized DataFrame ready for downstream processing.

	2.	feature_aggregator.py

	•	Purpose: Core time-window aggregation plus feature-time preprocessing.
	•	Main function/class: aggregate_to_windows(df, window_size='30T')
	•	Used by: Called by pipeline_controller.py after data_loader.py.
	•	Details:
	•	Windowing: assign window_id via time_utils.get_window_id() using 30-minute intervals.
	•	RF preprocessing: apply np.log1p(count) before aggregation (event level), then aggregate within each window.
	•	Grouping: groupby(['window_id','hostname']); produce per-window dict {rule_id: aggregated_count}.
	•	Output: Aggregated DataFrame [window_id, hostname, aggregated_rules_dict, (optional) label].

	3.	feature_generator.py

	•	Purpose: Convert aggregated rows into model-ready feature vectors.
	•	Main function/class: generate_feature_vectors(df_agg, mode)
	•	Used by: Called by pipeline_controller.py after feature_aggregator.py.
	•	Details:
	•	Pull ordered rule list and index map from rule_manager.py (e.g., 2898 rules).
	•	For each row, build dense vector from aggregated_rules_dict.
	•	If mode='train', append is_attack label.
	•	Output: X (feature matrix) and y (labels only in train mode).

B. Model Layer & Notifications
	1.	model_training.py

	•	Purpose: Train, tune (basic), and persist the model.
	•	Main function: train_model(X, y, config, model_save_path)
	•	Used by: pipeline_controller.py in mode='train'.
	•	Details:
	•	Split data (train_test_split).
	•	Model: RandomForestClassifier (RF).
	•	Scaling: not required for RF (keep optional scaler hooks for future models).
	•	Fit model; persist with joblib.dump().

	2.	model_evaluation.py

	•	Purpose: Evaluate trained model performance.
	•	Main function: evaluate_model(model, X_test, y_test, rule_list)
	•	Used by: Optionally called post-training by pipeline_controller.py, or run standalone.
	•	Details: Compute confusion matrix, classification report; extract feature importances.

	3.	model_predictor.py

	•	Purpose: Load trained artifacts and perform inference in detection mode.
	•	Main class: Predictor with .predict(X) / .predict_proba(X)
	•	Used by: pipeline_controller.py in mode='detect'.
	•	Details:
	•	Load model (joblib.load).
	•	Return structured prediction (label + confidence).

	4.	shap_explainer.py  

	•	Purpose: Generate local explanations for malicious predictions.
	•	Used by: Detection path only when the prediction is malicious; called before alerting.
	•	Details:
	•	Inputs: model, feature vector X_row, ordered rule_list/index map.
	•	Outputs: Top-N contributing rule IDs with SHAP values (and optional normalized weights).
	•	Integration: pass Top-N reasons to run_log.py and send_syslog.py.

	5.	send_syslog.py

	•	Purpose: Emit alerts to QRadar (syslog).
	•	Used by: Detection path after shap_explainer.py for malicious results.
	•	Details:
	•	Payload should include: hostname, predicted label, confidence, window_id, and Top-N SHAP reasons (rule IDs + contributions).
	•	Configurable facility/severity and endpoint from config.py.

C. Shared Utilities
	1.	rule_manager.py

	•	Purpose: Central management of the ordered rule universe (2898 IDs).
	•	Main functions: get_rule_list(), get_rule_to_index_map()
	•	Details: Load from rules.json/rules.txt; provide caching to avoid repeated I/O.

	2.	time_utils.py

	•	Purpose: All time handling utilities.
	•	Main functions: parse_qradar_timestamp(ts_string), get_window_id(timestamp, window_size)
	•	Used by: data_loader.py and feature_aggregator.py.

	3.	config.py

	•	Purpose: Centralized configuration (paths, DB URIs, model params, feature flags).
	•	Used by: Imported across the project (training/detection data paths; Mongo URI; thresholds; syslog target; Top-N for SHAP; etc.).

	4.	logging_utils.py / run_log.py

	•	Purpose: Unified logging across modules and explicit detection logging.
	•	Details:
	•	Provide helpers (e.g., log_pipeline_event, log_detection).
	•	In detection: log predicted label, confidence, and SHAP Top-N reasons alongside identifiers (hostname, window_id).

	5.	mongodb_connection.py

	•	Purpose: Central MongoDB management utility (connect, query, insert, delete).
	•	Used by: data_loader.py (mode='detect'), insert_DB.py, maintenance scripts.
	•	Details: Connection pooling, robust error handling, basic retries (as configured).

D. Data Acquisition & Management Jobs (QRadar → MongoDB)
	1.	create_searches_Qradar.py

	•	Purpose: Kick off QRadar searches on a schedule (e.g., every 30 minutes).
	•	Output: Search handles/IDs for tracking.

	2.	status_searches_Qradar.py

	•	Purpose: Poll search status until completion.
	•	Output: Completion flags/metadata for downstream retrieval.

	3.	result_searches_Qradar.py

	•	Purpose: Fetch completed search results from QRadar.
	•	Output: Raw result payloads for insertion.

	4.	insert_DB.py

	•	Purpose: Insert QRadar results into MongoDB.
	•	Used by: Follows result_searches_Qradar.py; uses mongodb_connection.py.
	•	Output: Normalized collections for the detection pipeline’s query window.

	5.	delete_searches_Qradar.py

	•	Purpose: Clean up completed QRadar searches (housekeeping).

	6.	delete_DB.py (MongoDB)

	•	Purpose: Scheduled database cleanup (e.g., daily) using mongodb_connection.py.
	•	Details: Enforce retention policy (TTL/roll-off) to control data volume.

E. Controller
	1.	pipeline_controller.py (or main.py)

	•	Purpose: Single entry point and orchestrator for both training and detection.
	•	Main function: run_pipeline(mode)
	•	Details:
	•	Parse CLI args (--mode train|detect).
	•	Load config.py.
	•	Training flow: data_loader → feature_aggregator → feature_generator → model_training → (optional) model_evaluation.
	•	Detection flow: data_loader → feature_aggregator → feature_generator → model_predictor → if malicious: shap_explainer → run_log → send_syslog.
	•	Centralized logging and exception handling.

Notes on Preprocessing & Consistency
	•	Log transform (np.log1p): performed in feature_aggregator.py before aggregation to shape count distributions consistently for both training and detection.
